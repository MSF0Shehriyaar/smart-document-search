## Abstract

This paper presents the design and implementation of **Smart Document Search**, a full-stack web application that enables users to upload PDF and text documents and perform intelligent search queries using the Term Frequency-Inverse Document Frequency (TF-IDF) algorithm. The system features a modern cyberpunk-inspired glassmorphism user interface built with React 19 and Tailwind CSS, an Express.js backend with in-memory document storage, and real-time relevance scoring with animated result visualization. The architecture is designed to be AI-ready, incorporating the Google GenAI SDK for future semantic search enhancements. This paper discusses the system architecture, the TF-IDF information retrieval methodology, implementation details, and potential future directions for AI-powered document analysis.

**Keywords:** Information Retrieval, TF-IDF, Document Search, React, Express, PDF Parsing, Full-Text Search, Vector Space Model

## 1. Introduction

### 1.1 Background

In the modern digital era, the volume of unstructured text data generated by individuals and organizations has grown exponentially. Documents such as PDFs, text files, and reports contain valuable information, but extracting relevant content efficiently remains a significant challenge. Traditional file-system searches rely on filename matching or basic string containment, which often fail to capture semantic relevance or rank results by importance.

Information Retrieval (IR) systems address this challenge by employing algorithms that measure the relevance between a user's query and the contents of a document collection. Among the foundational techniques in IR, the TF-IDF weighting scheme remains widely used due to its simplicity, interpretability, and effectiveness for many practical applications.

### 1.2 Problem Statement

Existing document search solutions often fall into two categories: (1) overly simplistic tools that lack ranking capabilities, or (2) complex enterprise search platforms that require significant infrastructure (databases, search engines like Elasticsearch, cloud services). There is a need for a lightweight, self-contained document search system that can run locally, process common document formats, and provide ranked search results without external dependencies.

### 1.3 Objectives

The primary objectives of this project are:

1. To develop a full-stack web application for uploading and searching PDF and text documents.
2. To implement a TF-IDF-based ranking algorithm for relevance scoring.
3. To design an intuitive, visually engaging user interface with real-time feedback.
4. To architect the system for future integration with Large Language Models (LLMs) and semantic search.

### 1.4 Scope

The current implementation supports:
- Multi-file upload of PDF and TXT documents
- In-memory storage and indexing
- Real-time TF-IDF search with relevance ranking
- Responsive glassmorphism UI with motion animations
- AI SDK integration readiness for future enhancements

## 2. Literature Review

### 2.1 Information Retrieval and the Vector Space Model

Information Retrieval (IR) is the science of searching for information in large collections of unstructured data [1]. The Vector Space Model (VSM), introduced by Gerard Salton in the 1970s, represents documents and queries as vectors in a high-dimensional space where each dimension corresponds to a term in the corpus [2]. The relevance between a query and a document is computed as a function of their vector representations.

### 2.2 TF-IDF Weighting

Term Frequency-Inverse Document Frequency (TF-IDF) is a numerical statistic that reflects how important a word is to a document in a collection [3]. It is the product of two metrics:

- **Term Frequency (TF):** Measures how frequently a term appears in a document.
- **Inverse Document Frequency (IDF):** Measures how rare a term is across the entire corpus.

The TF-IDF weight increases proportionally to the number of times a word appears in the document but is offset by the frequency of the word in the corpus, which helps to control for the fact that some words are generally more common than others.

### 2.3 Modern Web Architectures for IR Applications

Contemporary web applications for document processing typically employ a client-server architecture with a Single Page Application (SPA) frontend and a RESTful API backend [4]. React, combined with build tools like Vite, has become a standard for frontend development due to its component-based architecture and fast development experience [5]. On the backend, Node.js with Express provides a lightweight, event-driven runtime suitable for I/O-bound operations such as file uploads and text processing.

### 2.4 PDF Text Extraction

Extracting text from PDF documents is a non-trivial task due to the format's complex internal structure. Libraries such as `pdf-parse` (based on Mozilla's PDF.js) provide robust capabilities for rendering PDF content to plain text, enabling downstream text analysis and search operations [6].

## 4. Methodology

### 4.1 Document Ingestion Pipeline

The document ingestion process consists of the following steps:

1. **File Selection:** Users select one or more files through a drag-and-drop interface or file picker.
2. **Upload:** Files are transmitted to the server via HTTP POST with `multipart/form-data` encoding.
3. **Parsing:** The server inspects each file's MIME type and extension. PDF files are parsed using `pdf-parse`; text files are read directly as UTF-8 strings.
4. **Storage:** Extracted content is stored in an in-memory array alongside metadata (unique ID, filename, MIME type).
5. **Library Update:** The client fetches the updated document list and renders it in the sidebar.

### 4.2 TF-IDF Algorithm Implementation

The search algorithm implemented in the application follows the standard TF-IDF formulation with the following specifics:

#### 4.2.1 Tokenization

Text is tokenized using a regular expression that extracts word boundaries:

```
tokens = text.toLowerCase().match(/\b(\w+)\b/g)
```

This converts text to lowercase and extracts sequences of alphanumeric characters, filtering out punctuation and whitespace.

#### 4.2.2 Term Frequency (TF)

For a given term `t` and document `d`:

```
TF(t, d) = (frequency of t in d) / (total tokens in d)
```

This normalized frequency prevents bias toward longer documents.

#### 4.2.3 Inverse Document Frequency (IDF)

For a term `t` across corpus `D`:

```
IDF(t, D) = log((|D| + 1) / (documents containing t + 1)) + 1
```

The smoothing terms (`+1`) prevent division by zero and ensure that terms appearing in all documents still receive a non-zero weight.

#### 4.2.4 Scoring Function

The total relevance score for a document given a query is:

```
Score(q, d) = sum over all unique query terms t of [ TF(t, d) * IDF(t, D) ]
```

Documents with a score greater than zero are returned, sorted in descending order by score.

#### 4.2.5 Result Enrichment

Each result includes:
- Document ID and filename
- Computed relevance score (floating-point, 4 decimal places)
- A 200-character text snippet from the beginning of the document
- Match confidence percentage derived from the score

The UI follows a cyberpunk-inspired aesthetic with the following design principles:

## 5. Implementation

### 5.1 Backend Implementation (server.ts)

The Express server is written in TypeScript and provides three primary API endpoints:

#### POST /api/upload

Accepts an array of files via Multer's memory storage. Each file is processed based on its type:

- **PDF:** Parsed using `PDFParse` with error handling for invalid or corrupted files
- **Text:** Converted from buffer to UTF-8 string

Processed documents are pushed to the in-memory `documents` array with a randomly generated ID.

#### GET /api/documents

Returns a sanitized list of documents containing only `id`, `name`, and `type` fields, protecting the full text content from unnecessary exposure.

#### POST /api/search

Accepts a JSON body with a `query` string. The server:

1. Validates the query presence
2. Tokenizes the query and all document contents
3. Computes TF-IDF scores for each document
4. Filters results with score > 0
5. Sorts results by score descending
6. Returns enriched result objects

### 5.2 Frontend Implementation (App.tsx)

The React frontend is implemented as a single functional component with the following state management:

| State | Type | Purpose |
|-------|------|---------|
| `query` | string | Current search input |
| `results` | SearchResult[] | Ranked search results |
| `docs` | AppDoc[] | Uploaded document library |
| `isUploading` | boolean | Upload operation status |
| `isSearching` | boolean | Search operation status |
| `error` | string | Error message display |

The component is structured into three main regions:

1. **Header:** Application branding, system status, and engine state indicator
2. **Sidebar (Left):** File upload zone and active document library
3. **Main Content (Right):** Search input form and results display area

### 5.3 PDF Parsing Challenges

PDF parsing presents several challenges that the application addresses:

- **Invalid PDFs:** Caught and reported with descriptive error messages
- **Corrupted Files:** Detected via PDF.js exception types (`InvalidPDFException`, `FormatError`, `AbortException`)
- **Binary Content:** Non-PDF files with `.pdf` extensions are rejected with clear feedback

### 5.4 Error Handling

The application implements comprehensive error handling at multiple layers:

- **Network Errors:** Fetch failures are logged to the console with descriptive messages
- **Upload Errors:** Server-side parsing errors are propagated to the client UI
- **Validation Errors:** Missing files or queries return appropriate HTTP status codes (400 Bad Request)
- **Runtime Errors:** PDF parsing exceptions are caught and returned as JSON error responses

## 6. Results and Features

### 6.1 Functional Features

The implemented system provides the following capabilities:

1. **Multi-File Upload:** Users can upload multiple PDF and TXT files simultaneously
2. **Document Library:** A persistent sidebar displays all uploaded documents with file type icons
3. **Real-Time Search:** Queries are processed instantly against the in-memory index

## 7. Future Work

### 7.1 Semantic Search with Gemini Embeddings

The application includes the `@google/genai` SDK and Vite environment configuration for `GEMINI_API_KEY`. Future enhancements include:

- **Embedding Generation:** Using Gemini's `text-embedding` models to convert documents and queries into dense vector representations
- **Cosine Similarity:** Replacing TF-IDF with vector similarity for semantic matching beyond lexical overlap
- **Hybrid Scoring:** Combining TF-IDF and embedding scores for optimal relevance

### 7.2 AI-Generated Summaries

Integration with Gemini's generative models would enable:

- **Document Summarization:** Automatic generation of concise summaries for uploaded documents
- **Answer Extraction:** Direct answers to user questions with citation to source documents
- **Query Expansion:** AI-suggested query reformulations to improve recall

### 7.3 Persistent Storage

Current in-memory storage is ephemeral. Production deployment would require:

- **Database Integration:** PostgreSQL or MongoDB for document persistence
- **Vector Database:** Pinecone, Weaviate, or pgvector for embedding storage and approximate nearest neighbor search
- **Session Management:** User accounts and private document collections

### 7.4 Enhanced File Format Support

Future versions could support:

- Microsoft Word (.docx) documents
- Rich Text Format (.rtf)
- Markdown (.md) files
- Optical Character Recognition (OCR) for scanned PDFs

### 7.5 Advanced IR Techniques

- **BM25:** A probabilistic retrieval framework that often outperforms TF-IDF
- **Query Expansion:** Pseudo-relevance feedback to expand queries with related terms
- **Result Diversification:** Ensuring result sets cover multiple aspects of a query

## 8. Conclusion

Smart Document Search demonstrates that effective information retrieval systems can be built with modern web technologies without requiring heavy infrastructure. The TF-IDF algorithm, despite its age, provides robust and interpretable ranking for document search tasks. The combination of React's component model, Vite's fast development experience, and Express's simplicity creates a productive full-stack development environment.

The application's modular architecture and AI SDK integration position it well for evolution into a semantic search platform. The glassmorphism UI with motion design creates an engaging user experience that distinguishes it from traditional enterprise search interfaces.

This project serves as a foundation for understanding information retrieval concepts while providing a practical tool for personal document management and search.
